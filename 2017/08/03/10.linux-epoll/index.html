Linux内核如何高效处理海量网络IO?

发表于 2017-08-03

0x01.网络IO模型

前言

Linux由林纳斯·托瓦兹在1991年10月5日首次发布。系统使用宏内核负责处理进程控制、网络，以及外围设备和文件系统的访问。作为目前非常主流的服务端操作系统，它是如何应对海量的网络IO，并且高效的处理的?接下来我们从网络模型和内核IO调度等方面来剖析！

两大IO模型

概念

为什么我们要从网络IO模型开始说起，因为网络IO的本质其实是Socket的读写。当海量的请求到达服务器时，服务器如何高效的处理这些请求，我们需要一种高效处理的逻辑。这正是网络IO模型的作用之处，它是一种高效处理的逻辑，正是由于思想才锻造了内核处理海量IO并发的高效。
网络IO模型目前可以分为两大类同步模型、异步模型，两大模型下面又有各种的小分类。同步模型包括阻塞IO(Blocking IO)、非阻塞IO(Non-blocking IO)、多路复用IO(multiplexing IO)、信号驱动式IO(Signal-driven IO)。异步模型包括异步IO(Asynchronous IO)。下面我们解释这5种IO模型是如何工作的。

名词

首先我们先了解几个概念的词汇，待会解释IO模型是会用到。

文件描述符(file descriptor)简称fd
文件描述符在形式上是一个非负整数。实际上，它是一个索引值，指向内核为每一个进程所维护的该进程打开文件的记录表。当程序打开一个现有文件或者创建一个新文件时，内核向进程返回一个文件描述符。在程序设计中，一些涉及底层的程序编写往往会围绕着文件描述符展开。但是文件描述符这一概念往往只适用于UNIX、Linux这样的操作系统。当与服务端建立一个socket链接时，在socket上收发数据，相当于对一个文件进行读写，所以一个socket句柄，通常也用fd来表示。

用户空间和内核空间
Linux操作系统和驱动程序运行在内核空间，应用程序运行在用户空间，两者不能简单地使用指针传递数据。操作系统的核心是内核，独立于普通的应用程序，可以访问受保护的内存空间，也有访问底层硬件设备的所有权限。为了保证用户进程不能直接操作内核（kernel），保证内核的安全，操心系统将虚拟空间划分为两部分，一部分为内核空间，一部分为用户空间，供各个进程使用。

系统内核用户态和内核态
因为操作系统的资源是有限的，如果访问资源的操作过多，必然会消耗过多的资源，而且如果不对这些操作加以区分，很可能造成资源访问的冲突。所以，为了减少有限资源的访问和使用冲突，Unix/Linux的设计哲学之一就是：对不同的操作赋予不同的执行等级，就是所谓特权的概念。简单说就是有多大能力做多大的事，与系统相关的一些特别关键的操作必须由最高特权的程序来完成。Intel的X86架构的CPU提供了0到3四个特权级，数字越小，特权越高，Linux操作系统中主要采用了0和3两个特权级，分别对应的就是内核态和用户态。运行于用户态的进程可以执行的操作和访问的资源都会受到极大的限制，而运行在内核态的进程则可以执行任何操作并且在资源的使用上没有限制。很多程序开始时运行于用户态，但在执行的过程中，一些操作需要在内核权限下才能执行，这就涉及到一个从用户态切换到内核态的过程。比如C函数库中的内存分配函数malloc()，它具体是使用sbrk()系统调用来分配内存，当malloc调用sbrk()的时候就涉及一次从用户态到内核态的切换，类似的函数还有printf()，调用的是wirte()系统调用来输出字符串，等等。用户态和内核态之间的切换也会消耗大量资源。

详解

我们先从同步模型开始介绍
1.阻塞IO(Blocking IO)

应用进程调用系统函数recvfrom()从内核请求数据，内核如果没有数据将等待，直到网卡传递数据过来，内核开始将数据准备，数据准备完成后将数据从内核中拷贝到用户空间，这段时间用户进程处于阻塞状态。这种模式可以看出数据能够及时的返回，但是应用程序进程被阻塞了。

2.非阻塞IO(Non-blocking IO）

非阻塞IO中应用程序一直在轮询内核中是否有数据，直到内核缓冲区出现数据，然后等待数据从内核拷贝到用户空间，然后返回成功提示。该模式下应用进程是在轮询时是不阻塞的，但是数据复制的这段时间是阻塞的，而且该模式大量降低数据吞吐量，所以和阻塞模型中就出现了著名的C10K问题(处于此处的两个模型下内核当请求达到10k时，因为应用程序需要10k的线程进行处理，假设１个线程需要１Ｍ内存，10K的线程需要10G的内存，这在当时32位的操作系统来说完全不可能，因为32位的操作系统架构下，内存最多$２^{32}$，所以当时没法处理这种请求)。

3.多路复用IO(Multiplexing IO）

IO多路复用模型相对其他模型来说就高效多了，应用程序再也不用自己轮询或者等待内核数据，只需要把全量fd注册到select/poll中，然后让select/poll轮询内核中有哪些fd已经准备好数据，将准备好数据的fd数组返回。应用程序接受到数据后，相应的fd调用recvfrom()从内核高速缓存区将数据拷贝到应用缓存区。select/poll的原理就是遍历所有的fd的完成状态，当某个fd有数据到达时添加到可读数组，遍历完成后返回可读fd数组，通知应用程序来处理。这种看似非常优秀的设计也存在一定的问题。当fd数量巨大，十万、百万级别时，因为cpu将大量的fd从用户态复制到内核态进行遍历，将非常耗时，而且浪费系统资源。再复杂的问题都有解决方案，这时epoll模型(属于IO多路复用)就应运而生，这也是目前linux系统最为成熟的内核方案。这里不做赘述，后面我们将详细讲解。

4.信号驱动式IO(signal-driven IO)

信号IO模型，应用程序先到内核中注册一个sigio，当内核中有数据到到达，内核将回调信号通知应用程序。应用程序接到信号，调用recvfrom()进行数据接收。整个过程中信号建立到信号返回过程程序是不阻塞的，数据从内核中拷贝到用户空间时程序阻塞。

5.异步IO(Asynchronous IO)

异步IO模型，应用程序调用aio_read()，将描述文件符、缓冲区指针、缓冲区大小和文件偏移(aiocbp->aio_offset)传递给内核，内核操作完成将通知应用程序。该过程应用程序不阻塞。

同步IO模型其实都是阻塞的。从第一种阻塞IO开始到信号驱动IO，优化的是准备数据前的过程，但是从内核复制数据到用户空间这个过程IO还是阻塞的。异步IO模型中内核通过让应用程序注册缓存区指针方式，当数据准备完成，直接从内核拷贝到指定的用户空间，然后再通知应用程序，这样就能使应用程序一直处于非阻塞状态。当然并非异步IO模型就一定优于同步IO模型，目前linux上AIO模型并不是非常成熟（异步模型比较成熟的可以参考windows下的IOCP），同步模型epoll是目前linux上较为成熟的IO模型，高并发请求下表现良好。
linux能高效处理海量IO，也是由于高效的IO策略，但具体这些IO策略中有哪些可以学习的地方，我们继续来探寻目前比较成熟的epoll/aio模型的相关机制，了解下AIO模型。

0x02.Epoll、Aio

Epoll

于Linux 2.5.44首度登场的epoll是Linux内核的可扩展I/O事件通知机制[1]。它设计目的旨在取代既有POSIX select(2)与poll(2)系统函数，让需要大量操作文件描述符的程序得以发挥更优异的性能(举例来说：旧有的系统函数所花费的时间复杂度为O(n)，epoll的时间复杂度O(log n))。epoll与FreeBSD的kqueue类似，底层都是由可配置的操作系统内核对象建构而成，并以文件描述符(file descriptor)的形式呈现于用户空间。

关键方法

关于epoll,有三个关键方法我们先来了解下
1.int epoll_create(int size)
创建一个epoll句柄，size用来告诉内核这个监听的最大值(当size并非限制最大值，应该算一个初始最大值)，epoll创建会占用一个fd值，在使用完epoll后必须关闭，不然会造成fd泄露。

2. int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event)
epfd :　epoll_create的返回值。
op : op操作，添加EPOLL_CTL_ADD，删除EPOLL_CTL_DEL，修改EPOLL_CTL_MOD。分别添加、删除和修改对fd的监听事件。
fd : 需要监听的文件描述符。
epoll_event :　epoll监听的事件。

3. int epoll_wait(int epfd, struct epoll_event * events, int maxevents, int timeout)
等待epfd上的io事件，最多返回maxevents个事件。
参数events用来从内核得到事件的集合，maxevents告之内核这个events有多大，这个maxevents的值不能大于创建epoll_create()时的size，参数timeout是超时时间（毫秒，0会立即返回，-1将不确定，也有说法说是永久阻塞）。该函数返回需要处理的事件数目，如返回0表示已超时。

结构体代码

epoll_create会在linux内核会创建一个eventpoll结构体

每个epoll对象都有一个独立的eventpoll结构体，结构体中存放通过epoll_ctl向epoll对象中添加文件描述符(fd)，fd位于红黑树中，红黑树的插入时间效率是$lgn$。所有添加到epoll中的fd都会与网卡驱动程序建立回调关系。当IO事件发生时，这个回调方法会在内核中调用ep_poll_callback，改函数将fd添加到rdlist双链表中。

在epoll对象中，每个fd都会对应建立一个epitem结构体

epitem结构体中rbn是记录红黑树的节点，rdllink是记录双向链表的节点。

ET&LT

epoll_wait返回句柄操作中，有两种模式。一种模式为LT(Level Triggered 水平触发)、ET(Edge Triggered 边缘触发)
LT(Level Triggered 水平触发)
accept一个连接，添加到epoll中监听EPOLLIN事件，当EPOLLIN事件到达时，read fd中的数据并处理，当需要写出数据时，把数据write到fd中；如果数据较大，无法一次性写出，那么在epoll中监听EPOLLOUT事件，当EPOLLOUT事件到达时，继续把数据write到fd中；如果数据写出完毕，那么在epoll中关闭EPOLLOUT事件。
socket接收缓冲区不为空 有数据可读 读事件一直触发，socket发送缓冲区不满 可以继续写入数据 写事件一直触发，epoll_wait返回的事件就是socket的状态。

ET(Edge Triggered 边缘触发)
accept一个一个连接，添加到epoll中监听EPOLLIN|EPOLLOUT事件，当EPOLLIN事件到达时，read fd中的数据并处理，read需要一直读，直到返回EAGAIN为止，当需要写出数据时，把数据write到fd中，直到数据全部写完，或者write返回EAGAIN 当EPOLLOUT事件到达时，继续把数据write到fd中，直到数据全部写完，或者write返回EAGAIN。
socket的接收缓冲区状态变化时触发读事件，即空的接收缓冲区刚接收到数据时触发读事件，socket的发送缓冲区状态变化时触发写事件，即满的缓冲区刚空出空间时触发读事件，仅在状态变化时触发事件。

使用ET模式，特定场景下会比LT更快，因为它可以便捷的处理EPOLLOUT事件，省去打开与关闭EPOLLOUT的epoll_ctl（EPOLL_CTL_MOD）调用。从而有可能让你的性能得到一定的提升。例如你需要写出1M的数据，写出到socket 256k时，返回了EAGAIN，ET模式下，当再次epoll返回EPOLLOUT事件时，继续写出待写出的数据，当没有数据需要写出时，不处理直接略过即可。而LT模式则需要先打开EPOLLOUT，当没有数据需要写出时，再关闭EPOLLOUT（否则会一直返回EPOLLOUT事件）总体来说，ET处理EPOLLOUT方便高效些，LT不容易遗漏事件、不易产生bug。如果server的响应通常较小，不会触发EPOLLOUT，那么适合使用LT，例如redis等。而nginx作为高性能的通用服务器，网络流量可以跑满达到1G，这种情况下很容易触发EPOLLOUT，则使用ET。关于某些场景下ET模式比LT模式效率更好

高效总结

epoll相较之前的select、opll不需要将socket句柄(fd)反复传递给内核，epoll数据结构中红黑树持有所有的socket句柄(fd)，并且能高效进行增删查，当有数据进来并从网卡中负责到内核后，将对应的socket句柄(fd)添加双链表中，然后应用程序根据双链表中socket句柄(fd)进行对应的IO数据读取。
