0x01.Mybatis简介

MyBatis 是一款优秀的持久层框架，目前主流的ORM框架,它支持定制化 SQL、存储过程以及高级映射。MyBatis 避免了几乎所有的 JDBC 代码和手动设置参数以及获取结果集。MyBatis 可以使用简单的 XML 或注解来配置和映射原生信息，将接口和 Java 的 POJOs(Plain Old Java Objects,普通的 Java对象)映射成数据库中的记录,并且支持事物管理机制。功能非常强大,而且实现起来非常简单,优雅。

0x02.由浅入深

实例

平常我们在使用Mybatis时,或通过XML配置方式进行数据库访问,或通过Java API的方式进行访问,我们配置的XML或者编写的代码通过Mybatis是如何获取到数据库里的信息。我们先来写一个简单的Select例子,进行逐步分析。

首先我们创建一张User用户信息表,并插入数据,表信息如下:

id name age tel
1 tom 10 010-8000001
2 jack 21 13489832789
3 lilei 36 0571-88088221

创建Mapper XML配置文件
UserMapper.xml

UserMapper.xml中动态Sql是通过OGNL语法来实现(OGNL从sql参数对象中计算表达式的值，根据表达式的值动态拼接sql，以此来完成动态sql的功能,但是如果要写的完整,必须配合where,trim节点,where节点是判断包含节点有内容就插入where,否则不插入,trim节点是用来判断如果动态语句是以and或or开始,如果是会自动去掉开头and或者or)。

创建UserMapper接口
UserMapper.java

UserMapper接口,就是映射文件中的namespace的值。selectByAge接口的方法名，就是映射文件中MappedStatement的id值，接口方法内的参数，就是传递给sql的参数。Mapper接口是没有实现类的，当调用接口方法时，接口全限名+方法名拼接字符串作为key值,可唯一定位一个MappedStatement。
举例:com.java.example.Mybatis.UserMapper.UserMapper.selectByAge,可以唯一找到com.java.example.Mybatis.UserMapper.UserMapper下面id = selectByAge的MappedStatement。在Mybatis中，每一个< select >、< insert >、< update >、< delete >标签，都会被解析为一个MappedStatement对象。
UserMapper接口里的方法，是不能重载的，因为是全限名+方法名的保存和寻找策略。
UserMapper接口的工作原理是JDK动态代理，Mybatis运行时会使用JDK动态代理为Mapper接口生成代理proxy对象，代理对象proxy会拦截接口方法，转而执行MappedStatement所代表的sql，然后将sql执行结果返回。

通过实例看出我们只需要通过简单的配置xml文件,然后定义一个mapper接口,就能进行数据库的增删改查并且将结果进行封装成你需要的类型,Mybatis是如何做到这些,我们接下来继续分析。实例中我们通过xml绑定的方式实现接口绑定,当然你也可以通过注解绑定的方式去实现,一般在sql比较复杂的时候,我们使用xml绑定,sql比较简单的时候使用注解绑定(@Select@Update等)

探索究竟

Mybatis把sql语句从Java源程序中独立出来，放在单独的XML文件中编写，给程序的维护带来了很大便利。 Mybatis封装了底层JDBC API的调用细节，并能自动将结果集转换成Java Bean对象，大大简化了Java数据库编程的重复工作。因为Mybatis需要程序员自己去编写sql语句，程序员可以结合数据库自身的特点灵活控制sql语句，因此能够实现比Hibernate等全自动ORM框架更高的查询效率，能够完成复杂查询。那它是如何做到这些的呢? 我们自己实现查询数据库获取相应对象的功能,需要做哪些步骤。

  1. 首先加载JDBC驱动,建立并获取数据库连接
  2. 其次我们需要创建JDBC Statement对象
  3. 然后设置SQL语句并传入参数
  4. 接着执行SQL语句获取到执行结果
  5. 再对执行结果进行转换处理,返回相应的结果
  6. 最后释放相关资源（关闭Statement，关闭ResultSet，关闭Connection）

完成上述步骤后,我们就能跟上述实例中一样获取一个Java Bean对象,或者其他相应的结果。也就说mybatis帮我们完成了上述的大部分步骤,我们只需要操作SQL和Mapper就能获取到相应的结果,那到底mybatis到底是如何实现的呢,我们来进行深入探究。

我们先来看看mybatis的框架结构。

接口层处理,增删改查。数据处理层进行参数映射,SQL解析,执行SQL,映射结果。基础支撑层,提供最基础的数据库,缓存(包括一级缓存、二级缓存),事物等服务。
那具体到代码是如何实现的呢,我们结合Github上MyBatis源码分析。

参考之前实例中Main函数中方法实现。
SqlSessionFactoryBuilder类build方法创建sqlSessionFactory对象(SqlSessionFactory实例可以获得操作数据的SqlSession实例，通过这个实例对数据库进行操作),

sqlSessionFactory接口的openSession方法创建出一个SqlSession(SqlSession中包含了mybatis的事物控制和数据库会话),然后执行SqlSession中的selectList

selectList有个参数RowBounds,RowBounds对象进行分页，它是针对ResultSet结果集执行的内存分页，而非物理分页，可以在sql内直接书写带有物理分页的参数来完成物理分页功能，也可以使用分页插件Mybatis-PageHelper来完成物理分页(Mybatis仅可以编写针对ParameterHandler、ResultSetHandler、StatementHandler、Executor这4种接口的插件，Mybatis使用JDK的动态代理，为需要拦截的接口生成代理对象以实现接口方法拦截功能，每当执行这4种接口对象的方法时，就会进入拦截方法，具体就是InvocationHandler的invoke()方法，当然，只会拦截那些你指定需要拦截的方法,实现Mybatis的Interceptor接口并复写intercept()方法，然后在给插件编写注解，指定要拦截哪一个接口的哪些方法,配置文件中配置插件即可)
分页插件的基本原理是使用Mybatis提供的插件接口，实现自定义插件，在插件的拦截方法内拦截待执行的sql，然后重写sql，根据dialect方言，添加对应的物理分页语句和物理分页参数。
举例：select * from student，拦截sql后重写为：select t.* from （select * from student）t limit 0，10

Mapper.xml中的sql通过MappedStatement类中的getBoundSql转换成BoundSql(具体xml如何解析成SQL的可以参照Mybatis解析动态sql原理分析)。

最终通过SimpleExecutor的doQuery方法生成JDBC的statement。
Mybatis有三种基本的Executor执行器，SimpleExecutor、ReuseExecutor、BatchExecutor。

  1. SimpleExecutor：每执行一次update或select，就开启一个Statement对象，用完立刻关闭Statement对象。
  2. ReuseExecutor：执行update或select，以sql作为key查找Statement对象，存在就使用，不存在就创建，用完后，不关闭Statement对象，而是放置于Map<String, Statement>内，供下一次使用。简言之，就是重复使用Statement对象。
  3. BatchExecutr：执行update（没有select，JDBC批处理不支持select），将所有sql都添加到批处理中（addBatch()），等待统一执行（executeBatch()），它缓存了多个Statement对象，每个Statement对象都是addBatch()完毕后，等待逐一执行executeBatch()批处理。与JDBC批处理相同。

Executor的这些特点，都严格限制在SqlSession生命周期范围内。

执行完sql后返回的结果通过< resultMap >标签，逐一定义列名和对象属性名之间的映射关系。或者是使用sql列的别名功能，将列别名书写为对象属性名，比如T_NAME AS NAME，对象属性名一般是name，小写，但是列名不区分大小写，Mybatis会忽略列名大小写，智能找到与之对应对象属性名，你甚至可以写成T_NAME AS NaMe，Mybatis一样可以正常工作。
有了列名与属性名的映射关系后，Mybatis通过反射创建对象，同时使用反射给对象的属性逐一赋值并返回，那些找不到映射关系的属性，是无法完成赋值的。
